# AI 秋招八股文全面整理

## Agent相关


### 一、Agent 篇

#### 1.1 核心概念
**AI Agent** 是具备感知、思考、规划、工具调用、执行、记忆反思能力的智能体。区别于传统单纯问答的AI，它能自主拆解复杂任务、选择合适工具、通过多步骤执行，最终达成目标。

核心等式：**Agent = Model + Harness**
- Model：负责推理和生成，是"脑子"
- Harness：负责任务管理、上下文供给、工具反馈、验证机制、错误恢复，是"工作制度"

#### 1.2 核心六大组件
1. **大模型大脑**：推理决策核心
2. **感知输入**：接收用户指令、环境反馈
3. **规划模块**：目标拆解、路径选择
4. **记忆系统**：短期记忆（上下文）、长期记忆（向量库）
5. **工具调用**：API、数据库、文件系统等外部能力
6. **反思机制**：自我校验、错误修正

#### 1.3 主流推理范式
| 范式 | 核心思想 | 适用场景 |
|------|----------|----------|
| **CoT（思维链）** | 一步一步推理，输出中间思考过程 | 简单推理题、单步任务 |
| **ReAct** | Reasoning + Acting，思考→行动→观察循环 | 需要工具调用的任务 |
| **ToT（思维树）** | 多路径探索，可回溯、剪枝 | 复杂决策、多解空间问题 |
| **GoT（思维图）** | 图结构，支持节点聚合与分流 | 高度复杂的系统任务 |
| **Plan-and-Execute** | 先做完整规划，再分步执行 | 长链路、目标明确的任务 |

**ReAct 消息格式**（高频考点）：
- ``：模型内部思考
- `<tool_call>`：调用工具的结构化指令
- `<observation>`：工具执行返回的结果

#### 1.4 记忆系统三层架构
- **工作记忆**：当前对话上下文，存在context window中，容量最小
- **短期记忆**：本轮任务的历史执行记录，通常用滑动窗口管理
- **长期记忆**：跨任务的知识沉淀，通常存入向量数据库，按需检索

#### 1.5 高频面试题
**Q：ReAct 框架的工作原理是什么？**
A：ReAct 将推理（Reasoning）和行动（Acting）结合起来，形成"思考→行动→观察"的循环。模型先思考下一步该做什么，然后调用工具执行行动，获取观察结果，再基于结果继续思考，直到任务完成。核心是让推理过程可解释、行动有依据。

**Q：多智能体系统相比单 Agent 有什么优势？**
A：优势包括：1）专业分工，不同 Agent 擅长不同领域；2）并行处理，提升复杂任务效率；3）视角互补，减少单一模型偏见；4）容错性强，单个 Agent 失败不影响整体。但会引入协调成本、通信开销和一致性问题。

**Q：Agent 执行长链条任务时如何避免死循环？**
A：常用手段：1）设置最大迭代步数；2）引入状态检查点，检测重复状态；3）加入反思机制，让模型自我评估进展；4）超时熔断机制；5）人工干预入口。

### 四、RAG 篇

#### 4.1 核心概念
**RAG（Retrieval-Augmented Generation，检索增强生成）**：在 LLM 生成回答之前，先从外部知识库检索相关信息，把检索结果塞进 Prompt，让 LLM 基于事实回答。

**解决 LLM 三大痛点**：
1. **知识截止**：训练数据有截止日期，无法获取最新信息
2. **私有数据**：公司内部文档、客户数据无法进入训练集
3. **幻觉问题**：外部知识验证，降低编造概率

#### 4.2 RAG 完整链路
**索引阶段（离线）**：
1. 文档读取与解析（PDF/Word/Markdown/网页）
2. 文本切块（Chunking）——决定系统上限
3. Embedding 向量化
4. 存入向量数据库

**检索阶段（在线）**：
1. 用户问题输入
2. Query 改写（可选）
3. 多路召回：向量检索 + 关键词检索（BM25）
4. 重排序（Rerank）
5. 筛选最终文档片段

**生成阶段**：
1. 组装 Prompt：系统指令 + 检索上下文 + 用户问题
2. LLM 生成回答
3. 后处理：来源标注、格式化

#### 4.3 核心技术点
**Chunking 策略**：
- 固定长度切块：简单但语义断裂
- 语义感知切块：按语义边界切，效果更好
- 父子块（Parent-Child）：小子块检索精准，父块提供完整上下文
- 关键参数：chunk size（通常256-1024 token）、overlap（10%-20%）

**混合检索（Hybrid Search）**：
- 向量检索：捕捉语义相似度，擅长模糊匹配
- 关键词检索（BM25）：精确匹配术语、编号、实体
- 融合方式：RRF（倒数秩融合）、加权求和

**Query 改写**：
- Multi-Query：一个问题拆成多个角度子查询
- HyDE：先生成假设答案，用假设答案的向量去检索
- 问题补全：补全省略的上下文指代

**Rerank 重排序**：
- 解决"向量相似度 ≠ 答案相关性"
- 常用模型：BGE-Reranker、Cohere Rerank
- 先召回粗排（top-50），再重排精排（top-5）

#### 4.4 进阶方向
- **Agentic RAG**：Agent 自主判断是否需要检索、检索什么、如何验证
- **GraphRAG**：抽取实体关系构建知识图谱，支持多跳推理和全局总结
- **Multi-Modal RAG**：支持图像、表格、音视频等多模态检索

#### 4.5 高频面试题
**Q：RAG 和 Fine-Tuning 的区别？**
A：

| 维度 | RAG | Fine-Tuning |
|------|-----|-------------|
| 知识更新 | 动态更新，随时新增文档 | 静态，需重新训练 |
| 幻觉控制 | 较好，有来源可追溯 | 较差，模型内化后难验证 |
| 成本 | 低，主要是检索+推理 | 高，需要标注数据和算力 |
| 适用场景 | 事实问答、知识库、频繁更新 | 风格迁移、能力内化、响应速度要求高 |

**Q：RAG 效果不好怎么优化？**
A：从全链路排查：
1. **索引侧**：优化切块策略（语义切块、调整大小），提升 Embedding 质量
2. **检索侧**：改用混合检索，增加 Query 改写，加入 Rerank
3. **生成侧**：优化 Prompt 指令，强制引用标注，增加事实校验
4. **评估侧**：用 RAGAS 等框架量化评估，定位具体坏例针对性优化

**Q：GraphRAG 比普通 RAG 好在哪里？**
A：GraphRAG 擅长跨文档关系推理、多跳问题和全局性总结。普通向量RAG只能召回局部相似片段，对于"总结A公司和B公司的合作历程"这类需要串联多个文档的问题效果差。但 GraphRAG 构建和维护成本高，不是所有场景都需要。

### 五、Harness 篇

#### 5.1 核心定义
**Harness Engineering（线束工程）**：Agent 系统中，除模型本身之外，所有保障系统稳定可靠运行的工程结构的总称。

核心公式：**Harness = Agent − Model**

三者的包含关系：
- **Prompt Engineering**：对"指令"的工程化（最小范围）
- **Context Engineering**：对"输入环境"的工程化（包含 Prompt）
- **Harness Engineering**：对"整个运行系统"的工程化（包含前两者）

#### 5.2 Harness 四大核心职责
1. **执行调度**：控制执行顺序、并行度、条件路由
2. **监控可观测**：追踪执行链路、token消耗、成本统计
3. **错误隔离与恢复**：防止单点故障扩散，重试、降级、熔断
4. **上下文治理**：保证每一步模型拿到准确、适量、不被污染的上下文

#### 5.3 Harness 六层架构
1. **输入层**：任务规范化、意图识别、风险分级
2. **上下文层**：上下文组装、动态注入、压缩管理
3. **工具层**：工具注册、权限控制、执行沙箱
4. **状态层**：任务状态管理、检查点、持久化
5. **验证层**：输出校验、事实核查、约束检测
6. **观测层**：日志、指标、链路追踪、成本核算

#### 5.4 核心工程问题
**上下文污染**：历史对话中的错误信息、无关内容累积，影响后续推理
- 治理：定期上下文重置、关键信息提取摘要、分层注入

**代码熵积累**：Agent 多次修改代码后质量下降
- 治理：代码审查检查点、测试驱动执行、回滚机制

**工具调用可靠性**：API 超时、返回格式异常、权限不足
- 治理：超时设置、重试策略、Schema 校验、降级方案

#### 5.5 高频面试题
**Q：为什么说 Agent = Model + Harness？**
A：模型只负责推理生成，但真实生产环境中，任务管理、上下文组织、工具调度、错误处理、安全验证等大量工程问题，才是决定 Agent 能不能稳定交付的关键。模型好比脑子，Harness 是身体和管理制度，两者结合才是完整可用的 Agent。

**Q：Harness 和 Prompt Engineering 是什么关系？**
A：是包含关系。Prompt 只是 Harness 中上下文层的一小部分。Harness 覆盖了从输入接收、任务拆解、上下文管理、工具执行、结果验证到状态追踪的全链路，范围大得多。当任务还是单轮对话时 Prompt 就够，但进入长链路、可执行、低容错的生产场景，Harness 必不可少。

**Q：模型能力变强了，Harness 还重要吗？**
A：依然重要，但形态会变化。模型能力提升后，一些过去需要 Harness 补丁解决的问题（比如简单的格式控制）模型自己就能做好。但系统级问题——安全边界、成本控制、可观测性、故障隔离——永远需要工程层保障。而且模型越强，能做的任务越复杂，对 Harness 的要求反而越高。

### 六、Skill 篇

#### 6.1 核心定义
**Skill（技能）**：一种标准化的自然语言指令文件（通常是 SKILL.md），用来教 Agent"在什么场景下、按照什么方法、遵循什么规范来完成特定任务"。

类比：工具是"锤子、扳手"，Skill 是"维修手册"——告诉你什么时候用什么工具、按什么步骤操作。

#### 6.2 Skill 的典型结构
```yaml
---
name: code-review
description: 代码审查技能
trigger: 当用户要求进行代码审查时
dependencies: [filesystem, git]
---

# 代码审查规范

### 审查流程
1. 先读取完整文件，理解整体架构
2. 检查代码风格是否符合团队规范
3. 检查潜在bug和边界条件
4. 给出具体可执行的改进建议

### 注意事项
- 每次审查不超过5个核心问题
- 区分严重等级：阻断/重要/建议
- 给出具体代码示例，不说空话
```

#### 6.3 Skill 的核心价值
1. **知识沉淀**：把专家经验写成可复用的工作流
2. **一致性**：确保 Agent 每次处理同类任务都遵循相同标准
3. **可组合**：多个 Skill 可以组合处理复杂任务
4. **零代码扩展**：无需写代码，用自然语言就能扩展 Agent 能力

#### 6.4 Skill vs MCP vs Function Call
| 层级 | 解决的问题 | 通俗类比 |
|------|-----------|----------|
| **Function Call** | 让模型能调用单个函数 | 单个工具（一把螺丝刀） |
| **MCP** | 标准化工具接入协议，跨框架复用 | 统一插座标准（USB接口） |
| **Skill** | 定义任务的执行方法和流程规范 | 操作说明书（维修手册） |

**协作关系**：Skill 负责流程编排，不实现具体执行能力；MCP Tools 负责能力执行，不关心业务逻辑。Agent 自动识别 Skill 依赖的工具，匹配已启用的 MCP 工具完成调用。

#### 6.5 高频面试题
**Q：有了 MCP 为什么还需要 Skill？**
A：MCP 解决的是"能不能连上工具"的问题，Skill 解决的是"会不会用工具干活"的问题。就像给你一整套手术器械（MCP），你还是不会做手术，因为你不知道操作流程（Skill）。MCP 让 Agent 有了手，Skill 教 Agent 怎么用手完成具体工作。

**Q：Skill 和 Prompt 有什么区别？**
A：1）触发方式不同：Prompt 是每次手动写，Skill 是按触发条件自动激活；2）结构化程度不同：Skill 有标准格式（元数据+指令体），可管理可复用；3）粒度不同：Skill 是完整的任务方法论，Prompt 通常是单次任务指令；4）生态不同：Skill 可以安装、分享、版本化管理。

### 七、Claw 与 Claude 篇

#### 7.1 核心概念区分
**Claude**：Anthropic 开发的大语言模型系列（Opus/Sonnet/Haiku），是"大脑"。

**Claude Code**：Anthropic 官方推出的 CLI 编程助手，基于 Claude 模型，定位是"AI 结对编程工程师"。
- 专注代码开发全流程：生成、审查、重构、调试
- 深度集成文件系统、Git、终端
- 官方支持，安全可靠
- 只支持 Claude 模型

**OpenClaw**：开源的通用 AI Agent 框架/操作系统，定位是"全能自动化平台"。
- 多模型支持：可切换 Opus/Sonnet/Haiku，甚至接入其他厂商模型
- 多渠道接入：Telegram、Discord、Slack、WhatsApp
- 支持定时任务、心跳机制、长时运行
- 有技能市场（ClawMart）
- 支持多智能体编排

**ClaudeClaw**：轻量级开源包装器，在 Claude Code 之上模拟 OpenClaw 的行为，属于中间态产品。

#### 7.2 核心定位对比
| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| **定位** | 专精编程的AI助手 | 通用自动化Agent平台 |
| **模型支持** | 仅Claude系列 | 多模型路由 |
| **运行模式** | 任务型会话 | 7×24小时自主运行 |
| **外部渠道** | 无（终端/IDE内） | 丰富的IM集成 |
| **定时任务** | 不支持 | 原生支持cron |
| **Skill理念** | Skill=可调用工具 | Skill=可读说明书 |
| **开源性** | 闭源官方产品 | 完全开源 |
| **部署方式** | 本地CLI | Docker容器化 |
| **适用场景** | 开发者写代码 | 个人自动化、工作流全场景 |

#### 7.3 技术路线分歧
**Claude Code 路线**：
- Skill 视作工具（Tool），模块化可调用
- 强调产品化、易用性、安全性
- 官方把控生态质量
- 适合标准化交付场景

**OpenClaw 路线**：
- Skill 视作运行手册，模型自主阅读执行
- 强调开放性、扩展性、系统级架构
- 社区驱动生态
- 适合深度定制、本地化控制的企业场景

#### 7.4 高频面试题
**Q：Claude 和 OpenClaw 是什么关系？**
A：不是竞品，是互补关系。Claude 是模型（大脑），OpenClaw 是 Agent 平台（身体）。很多 OpenClaw 用户用 Claude 作为主力模型，Claude 负责思考，OpenClaw 负责连接外部系统、执行操作。

**Q：Claude Managed Agents 是什么？**
A：Anthropic 官方推出的云端托管 Agent 服务，对标 OpenClaw。云端托管、零运维、企业级安全，内置沙箱、代码执行、文件系统、网页浏览。但只支持 Claude 模型，按量付费。OpenClaw 是开源本地部署，灵活度更高。

### 八、MCP 篇

#### 8.1 核心定义
**MCP（Model Context Protocol，模型上下文协议）**：由 Anthropic 提出的开放协议，标准化 AI 模型接入外部工具和数据源的方式。

通俗类比：大模型界的 USB Type-C 接口。只要遵循 MCP 协议，模型就能像插 U 盘一样连接各种外部系统。

#### 8.2 三层架构
| 角色 | 职责 | 示例 |
|------|------|------|
| **MCP Host** | AI应用程序，协调模型、管理所有连接 | Claude Desktop、Cursor、你的Agent程序 |
| **MCP Client** | Host内部组件，负责与单个Server通信 | 嵌入在Host中的MCP客户端库 |
| **MCP Server** | 独立进程，对外提供工具/资源/提示词 | GitHub Server、Postgres Server、文件系统Server |

- 一个 Host 可以创建多个 Client
- 一个 Client 对应连接一个 Server
- Server 与 Client 是 1:1 关系

#### 8.3 核心能力原语
MCP 协议定义了三类核心能力：

1. **Tools（工具）**：模型可以调用的操作
   - `tools/list`：列出可用工具
   - `tools/call`：调用指定工具
   - 示例：执行SQL、发送邮件、调用API

2. **Resources（资源）**：模型可以读取的数据
   - `resources/list`：列出可用资源
   - `resources/read`：读取资源内容
   - 示例：文件内容、数据库表结构、文档

3. **Prompts（提示词模板）**：预定义的指令模板
   - `prompts/list`：列出可用提示词
   - `prompts/get`：获取提示词模板
   - 示例：代码审查模板、数据分析模板

#### 8.4 技术实现
- **通信协议**：基于 JSON-RPC 2.0
- **传输方式**：
  - STDIO：本地 Server，轻量高效
  - SSE（Server-Sent Events）+ HTTP：远程 Server
- **有状态会话**：支持订阅、通知、双向通信

#### 8.5 MCP vs Function Call
| 维度 | Function Call | MCP |
|------|--------------|-----|
| **性质** | 模型能力/API格式 | 开放协议标准 |
| **绑定** | 厂商绑定（OpenAI/Anthropic各自定义） | 跨厂商、跨框架 |
| **发现机制** | 静态配置，写死在代码里 | 动态发现，运行时获取工具列表 |
| **生态** | 各框架各自实现 | 一次编写，处处可用 |
| **能力范围** | 只有工具调用 | 工具+资源+提示词 |

#### 8.6 MCP vs A2A
- **MCP**：Agent ↔ 工具的连接标准，解决"触角能不能伸出去"的问题，请求-响应模式
- **A2A（Agent-to-Agent）**：Agent ↔ Agent 的协作标准，解决"脑子之间怎么配合"的问题，对等协商模式

#### 8.7 高频面试题
**Q：MCP 解决了什么痛点？**
A：在 MCP 之前，每个 AI 应用接外部工具都要单独写适配代码，工具生态碎片化。MCP 提供了统一标准，一个 MCP Server 开发完成后，所有支持 MCP 的客户端（Claude Desktop、Cursor、LangChain 等）都能直接使用，大大降低了集成成本，促进了工具生态的繁荣。

**Q：MCP 的 Server-First 设计是什么意思？**
A：传统 Function Call 是客户端定义工具，服务端（模型）被动调用。MCP 反过来，Server 主动声明自己有什么能力，Client 动态发现并使用。这样工具的维护者只需要管好 Server 端，不需要为每个客户端做适配，工具迭代更快。

**Q：为什么 MCP 要区分 Tools 和 Resources？**
A：因为两者语义不同。Tools 是"动作"，有副作用（比如删文件、发邮件），需要用户授权；Resources 是"数据"，只读操作（比如读文件、查文档），相对安全。分开定义可以做更精细的权限控制和安全策略。







## 大模型相关

### 一、大模型核心通用知识点
#### 1.1 主流架构分类
大语言模型按架构分为三类，当前工业界主流为**Decoder-only**架构：

| 架构类型 | 代表模型 | 核心特点 | 适用场景 |
|----------|----------|----------|----------|
| **Encoder-only** | BERT、RoBERTa | 双向注意力，理解能力强 | 分类、召回、语义匹配 |
| **Encoder-Decoder** | T5、BART、原Transformer | 编码+解码，跨序列对齐 | 翻译、摘要、结构化生成 |
| **Decoder-only** | GPT、LLaMA、Qwen、Claude | 自回归生成，因果掩码 | 对话、创作、通用大模型 |

#### 1.2 核心组件详解
###### （1）位置编码
解决Transformer本身无位置感知能力的问题，主流方案：
- **绝对位置编码**：直接给每个位置分配可学习向量，简单但泛化差，无法处理超过训练长度的序列。
- **正弦位置编码**：原Transformer方案，用三角函数生成固定编码，可外推但长序列精度下降。
- **RoPE（旋转位置编码）**：主流LLaMA系方案，通过旋转矩阵对Query和Key施加相对位置信息，兼具外推性和精度，支持线性内插、NTK缩放等长文本扩展。
- **ALiBi**：无显式位置向量，直接在注意力分数上加位置偏置，长文本外推能力极强。

###### （2）归一化方案
- **Post-Norm**：原Transformer方案，残差连接后做归一化。训练不稳定，深层模型易梯度消失。
- **Pre-Norm**：现代LLM标准方案，子层输入前先做归一化。训练更稳定，梯度传播更顺畅。
- **LayerNorm**：对每个样本的特征维度做归一化，不受batch大小影响，NLP场景标配。
- **RMSNorm**：LayerNorm的简化版，只做方差归一化，去掉均值中心化，速度更快，效果相当，LLaMA、Qwen等均采用。

###### （3）激活函数与FFN
- **FFN结构**：标准形式 `FFN(x) = Linear2(Activation(Linear1(x)))`，维度先升后降（通常4倍膨胀）。
- **主流激活函数演进**：
  - ReLU：简单但存在神经元死亡问题
  - GELU：高斯误差线性单元，BERT/GPT-2使用，平滑非单调
  - **SwiGLU**：门控线性单元，FFN拆成两路，一路激活一路直连，现代LLM标配，效果显著优于ReLU/GELU
- FFN参数量占比：通常占模型总参数量的70%左右，是参数量大头。

###### （4）注意力机制变种
- **Multi-Query Attention (MQA)**：多个头共享一组K/V，大幅降低显存占用，推理速度快，精度略有损失。
- **Grouped-Query Attention (GQA)**：折中方案，头分组，每组共享K/V，平衡速度和精度，LLaMA-2、Qwen等采用。

#### 1.3 大模型训练全流程
1. **预训练（Pre-training）**
   - 海量通用文本语料（万亿token级别），自监督语言建模（下一词预测）
   - 目标：让模型掌握通用知识、语法逻辑、世界规律
   - 特点：算力消耗最大，决定模型能力上限

2. **监督微调（SFT, Supervised Fine-Tuning）**
   - 高质量对话/指令数据（十万~百万级），教模型按照人类指令格式输出
   - 目标：对齐指令意图，学会对话格式，提升响应质量
   - 特点：数据质量 > 数据数量

3. **对齐训练（Alignment）**
   - 基于人类偏好进一步优化，让输出更符合人类价值观、更有用、更安全
   - 主流方案：RLHF、DPO、ORPO、KTO等
   - 目标：提升有用性、安全性、事实准确性

#### 1.4 训练优化核心技术
- **ZeRO（Zero Redundancy Optimizer）**：分布式训练显存优化方案，分三个等级：
  - ZeRO-1：切分优化器状态
  - ZeRO-2：切分优化器状态+梯度
  - ZeRO-3：切分优化器状态+梯度+模型参数，可训超大规模模型
- **梯度累积**：多步累积梯度后再更新，模拟大batch效果，解决显存不足
- **混合精度训练（AMP）**：前向用FP16/BF16加速，反向参数更新保留FP32，兼顾速度和精度
- **FlashAttention**：重排注意力计算顺序，利用SRAM减少显存读写，大幅提升速度并降低显存峰值

#### 1.5 推理优化核心技术
- **KV Cache**：缓存历史token的Key和Value，避免重复计算，生成阶段速度提升显著，是推理标配。
- **量化技术**：4bit/8bit量化，通过降低权重位宽减少显存占用，主流方案GPTQ、AWQ、FP8。
- **投机解码（Speculative Decoding）**：用小模型先生成候选token，大模型一次性验证，在不损失精度前提下提升解码速度。
- **批处理（Batching）**：连续批处理（Continuous Batching），动态管理请求队列，提升吞吐。

#### 1.6 高频面试题
**Q：为什么大模型都用Decoder-only而不是Encoder-Decoder？**
A：1）自回归生成天然适配Decoder架构，单向因果掩码符合逐token生成逻辑；2）训练更简单稳定，只有一种注意力模式；3）海量预训练数据下，Decoder-only在生成任务上效果更好，且零样本能力更强；4）工程优化成熟，KV缓存、并行推理等方案完善。

**Q：Pre-Norm 为什么比 Post-Norm 训练更稳定？**
A：Post-Norm把归一化放在残差之后，深层网络中残差路径的信号会被不断放大，导致梯度爆炸；Pre-Norm把归一化放在子层输入前，残差路径保持恒等映射，梯度传播更顺畅，深层模型更容易训练收敛。

**Q：KV Cache 为什么能加速推理？有什么代价？**
A：自回归生成时，每步只新增一个token，但如果没有缓存，每步都要重新计算所有历史token的K/V。KV Cache保存历史计算结果，每步只计算新增token的K/V，计算量从O(n²)降到O(n)。代价是需要额外显存存储K/V矩阵，序列越长显存占用越高。

### 二、LLaVA 与 LoRA 专项
#### 2.1 LLaVA 多模态大模型核心架构
LLaVA 是最经典的开源视觉语言大模型（VLM），核心思想是**用一个轻量投影层把视觉特征对齐到LLM的输入空间**，复用成熟的纯文本大模型能力。

**三段式架构**：
1. **视觉编码器**：提取图像特征
   - 通常用 CLIP ViT-L/14 作为骨干，也有用SigLIP、DinoV2的版本
   - 输入图像切分为patch，输出一组视觉token序列
   - 大多数训练阶段冻结，不参与微调

2. **投影层（Projection Layer）**：模态对齐的核心
   - 作用：把视觉特征维度映射到与LLM词嵌入相同的维度
   - 最简单是单层线性层，进阶版用2层MLP，对齐效果更好
   - 参数量极小，是第一阶段训练的主要对象

3. **大语言模型（LLM）**：负责推理和生成
   - 早期用Vicuna，现在主流用LLaMA、Qwen、Mistral等
   - 接收 <图像token序列 + 文本指令> 混合输入，自回归输出回答

#### 2.2 LLaVA 三阶段训练流程
1. **第一阶段：特征对齐预训练**
   - 冻结视觉编码器 + 冻结LLM，只训练投影层
   - 数据：图文对数据（如CC3M、LAION），任务是简单的描述生成
   - 目标：让视觉特征空间和文本词嵌入空间初步对齐

2. **第二阶段：端到端指令微调**
   - 冻结视觉编码器，微调投影层 + LLM（通常用LoRA）
   - 数据：高质量视觉指令跟随数据（LLaVA-Instruct-150K），覆盖问答、推理、细节描述
   - 目标：让模型学会遵循指令进行视觉问答

3. **第三阶段：多轮对话微调（可选）**
   - 用多轮对话数据进一步微调，提升多轮交互能力
   - 引入系统提示词、多轮上下文，优化对话连贯性

#### 2.3 LoRA 核心原理与数学推导
**LoRA（Low-Rank Adaptation，低秩适配）**：参数高效微调（PEFT）的主流方案，核心思想是**大模型权重矩阵的更新量是低秩的**，不用全量微调，只训练两个低秩矩阵。

**核心公式**：
```
W = W₀ + B·A
```
- `W₀`：原始预训练权重，冻结不更新
- `A`：降维矩阵，维度 d×r，用高斯初始化
- `B`：升维矩阵，维度 r×d，初始化为0
- `r`：秩（rank），远小于 d，通常取8/16/32/64

**前向计算**：
```
h = W₀·x + B·A·x
```
初始化时B为0，所以初始输出和原模型完全一致，训练稳定。

**参数量计算**：
- 原权重参数量：d²
- LoRA参数量：2 × r × d
- 当r=8、d=4096时，参数量仅为原权重的 0.4%

#### 2.4 LoRA 关键特性
1. **无推理额外延迟**：训练完成后可把BA合并回W₀，推理时和原模型结构完全一致，没有任何额外计算。
2. **插拔式**：一个基础模型可以挂载多个LoRA权重，切换不同任务/风格，无需保存多个全量模型。
3. **训练显存低**：不需要计算大部分参数的梯度和优化器状态，显存占用大幅降低。
4. **作用层选择**：通常只作用于注意力的Q、V投影矩阵，也可以扩展到Q/K/V/O和FFN；只训QV效果足够，加更多层收益递减。

#### 2.5 LoRA 主流变种
###### （1）QLoRA
- 核心：把基础模型量化到4bit，LoRA部分保持FP16/BF16训练
- 三大关键技术：
  1. 4bit NormalFloat (NF4) 量化：比普通4bit精度更高
  2. 双量化（Double Quantization）：量化常数本身再量化，进一步省显存
  3. 分页优化器（Paged Optimizers）：利用CPU内存做显存溢出，避免OOM
- 意义：单卡24G显存就能微调70B模型，大幅降低微调门槛

###### （2）DoRA（Weight-Decomposed LoRA）
- 核心：把权重分解为**幅度（magnitude）**和**方向（direction）**，LoRA只更新方向，幅度单独训练
- 解决传统LoRA：无法有效调整权重幅度，高秩下收益饱和的问题
- 效果：相同参数量下效果优于标准LoRA，更接近全量微调

###### （3）LoRA+
- 核心：给A矩阵和B矩阵设置不同学习率，B用更大的学习率
- 依据：B矩阵对最终输出影响更大，需要更大更新步长
- 几乎零成本提升训练效果和收敛速度

#### 2.6 多模态场景下的LoRA微调策略
以LLaVA为例，常见微调配置：
- **视觉编码器**：通常冻结，高级方案可以给ViT也加LoRA（只训最后几层）
- **投影层**：参数量小，一般全量微调
- **LLM部分**：主体用LoRA，秩r=16~128，作用于Q/K/V/O/FFN
- **训练策略**：分层学习率，投影层 > LoRA层 > 视觉编码器

#### 2.7 高频面试题
**Q：LoRA 为什么只更新Q和V就够了？**
A：注意力机制中，Q和V是信息变换的核心，决定了查询模式和值的表达；K和Q对称，O是最终融合。实验表明只微调Q和V就能达到不错的效果，继续加K/O/FFN收益递减。实际工程中会根据任务调整，复杂任务通常会扩大作用范围。

**Q：LoRA 的秩 r 怎么选？越大越好吗？**
A：不是越大越好。r太小表达能力不足，r太大参数量增加且容易过拟合。通常8~64是常用区间，r超过128后效果提升很有限。简单任务用小r（4~8），复杂任务用大r（32~64）。

**Q：QLoRA 会不会损失精度？和全量微调比怎么样？**
A：4bit量化会有轻微精度损失，但NF4量化方案已经把损失降到很低。在大多数下游任务上，QLoRA效果接近全量微调，远好于纯零样本。它的核心价值是用极低的显存成本达到接近全量微调的效果，是工业界微调的主流方案。

**Q：LLaVA 里投影层的作用是什么？能不能去掉？**
A：不能去掉。视觉编码器输出的特征维度和语义空间，与LLM的词嵌入空间不匹配，投影层就是做模态桥接，把视觉特征"翻译"成LLM能理解的表示。如果直接把视觉特征输入LLM，模型无法解读，效果会极差。

### 三、强化学习（含大模型对齐）知识点
#### 3.1 基础概念与MDP
###### 核心基础概念
- **智能体（Agent）**：做决策的主体（大模型）
- **环境（Environment）**：智能体交互的对象
- **状态（State, s）**：环境当前的局面
- **动作（Action, a）**：智能体做出的选择
- **奖励（Reward, r）**：环境反馈的标量信号，是优化目标
- **轨迹（Trajectory）**：状态-动作-奖励的序列 `s₀,a₀,r₁,s₁,a₁,r₂...`
- **折扣因子（γ）**：0~1之间，远期奖励打折扣，体现"即时奖励更重要"
- **回报（Return, G）**：一条轨迹的累计折扣奖励 `G = R₁ + γR₂ + γ²R₃ + ...`

###### 马尔可夫决策过程（MDP）
强化学习的数学框架，五元组 `(S, A, P, R, γ)`：
- S：状态空间
- A：动作空间
- P：状态转移概率 P(s'|s,a)
- R：奖励函数 R(s,a)
- γ：折扣因子

**贝尔曼方程**：核心递推关系
- 状态价值函数 V(s)：从状态s出发，能获得的期望回报
  `V(s) = E[ R + γ·V(s') ]`
- 动作价值函数 Q(s,a)：在状态s执行动作a后的期望回报
  `Q(s,a) = E[ R + γ·max_a' Q(s',a') ]`

#### 3.2 强化学习算法分类
| 类别 | 核心思想 | 代表算法 | 特点 |
|------|----------|----------|------|
| **基于值** | 学习价值函数，间接得到策略 | Q-Learning、DQN | 离散动作强，连续动作难 |
| **基于策略** | 直接优化策略函数 | Policy Gradient、REINFORCE | 连续动作友好，方差大 |
| **Actor-Critic** | 同时学策略和价值函数，互相迭代 | A2C、PPO、SAC | 平衡稳定与效率，工业界主流 |

#### 3.3 PPO 算法核心（大模型RLHF标配）
**PPO（Proximal Policy Optimization，近端策略优化）**：当前最主流的强化学习算法，也是RLHF的核心。
核心目标：**在保证策略更新幅度不要太大的前提下，最大化奖励**，既保证稳定又保证效率。

###### 三大损失函数
1. **策略损失（Clip目标）**
   ```
   L_CLIP = E[ min( r_t(θ)·A_t , clip(r_t(θ), 1-ε, 1+ε)·A_t ) ]
   ```
   - `r_t(θ)`：新旧策略的概率比值
   - `A_t`：优势函数（动作比平均水平好多少）
   - ε通常取0.2，限制策略更新幅度，防止步子太大崩掉

2. **价值损失（Value Loss）**
   - Critic网络的均方误差损失，拟合状态价值
   - 用来计算优势函数，指导策略更新

3. **熵奖励（Entropy Bonus）**
   - 给策略熵加正奖励，鼓励探索，防止过早收敛到确定性策略

###### PPO 核心优势
- 实现简单，超参数少，鲁棒性强
- 可以重复利用样本，样本效率高
- 天然适配大模型这种大规模网络训练

#### 3.4 大模型对齐：RLHF 完整流程
**RLHF（Reinforcement Learning from Human Feedback）**：用人类偏好数据训练奖励模型，再用强化学习让模型对齐人类偏好。

**三步标准流程**：
1. **第一步：SFT 监督微调**
   - 用高质量标注数据微调基础模型，得到SFT模型
   - 作用：让模型先学会基本的指令跟随格式

2. **第二步：奖励模型（RM）训练**
   - 收集人类偏好数据：同一个prompt，模型生成多个回答，人类排序好坏
   - 用排序对训练奖励模型：输入prompt+回答，输出一个标量奖励分
   - 损失：Pairwise Ranking Loss，让好回答得分高于坏回答

3. **第三步：PPO 强化学习微调**
   - 把奖励模型作为环境奖励信号
   - 用PPO算法微调SFT模型，最大化奖励分数
   - 通常加KL散度惩罚，防止模型偏离SFT太远、出现奖励黑客（Reward Hacking）

**三个模型协同**：
- 策略模型（Actor）：被优化的大模型
- 价值模型（Critic）：估计状态价值
- 参考模型（Reference）：SFT模型，计算KL惩罚

#### 3.5 免强化学习对齐方案
###### （1）DPO（Direct Preference Optimization）
- 核心思想：**跳过奖励模型和PPO，直接用偏好对优化策略**
- 数学上证明了：在最优策略下，奖励和策略满足一一对应关系，可以把奖励最大化问题直接转化为策略的分类损失
- 优势：
  - 实现简单，只需要监督训练，不需要强化学习复杂的调参
  - 训练稳定，没有PPO的超参数和不稳定性
  - 显存占用低，训练速度快
- 局限：复杂奖励信号建模能力弱于完整RLHF

###### （2）ORPO / KTO
- ORPO：把SFT和偏好对齐合并成一步，不用分开训练
- KTO：不需要成对偏好，只要知道"好/坏"二分类标签就能训

#### 3.6 高频面试题
**Q：PPO 为什么要加Clip？**
A：因为策略梯度方法中，如果新旧策略差异太大，梯度估计就不准了，容易出现训练崩溃。Clip机制强制限制策略更新幅度，让比值在[1-ε, 1+ε]区间内，保证每次更新都是小步走，训练更稳定。

**Q：RLHF 为什么要加KL惩罚？**
A：两个作用：1）防止模型为了刷奖励分而输出奇怪的文本（奖励黑客），保持模型的自然语言能力；2）约束优化空间，让训练更稳定，避免策略偏离SFT基线太远。本质是用性能换稳定性和安全性。

**Q：DPO 和 PPO 相比有什么优缺点？**
A：
- DPO优势：实现简单、训练稳定、算力消耗低、调参成本低
- DPO劣势：只能处理成对偏好，无法建模复杂的多步奖励，扩展性不如PPO；极端场景下对齐深度可能不如RLHF
- 适用场景：数据简单、追求快速落地用DPO；复杂对齐、工业级大规模用PPO/RLHF

**Q：奖励模型为什么不直接用人类打分，要用排序？**
A：人类的绝对分数不稳定，不同人打分尺度不一样，甚至同一个人前后标准也不一致。但人类做"哪个更好"的相对排序要准确得多，一致性更高。所以用成对排序训练奖励模型，标注质量更高，数据更可靠。





## 项目相关

### Transformer 整体架构
原始 Transformer 为 Encoder-Decoder 结构，各堆叠 6 层。

**Encoder 每层（2个子层）：**
1. Multi-Head Self-Attention
2. Feed Forward Network (FFN)
- 每个子层都有残差连接 + LayerNorm

**Decoder 每层（3个子层）：**
1. Masked Multi-Head Self-Attention（因果掩码，防止看未来token）
2. Cross-Attention（Q来自Decoder，K/V来自Encoder输出）
3. Feed Forward Network (FFN)
- 每个子层同样有残差连接 + LayerNorm

**输入端**：Token Embedding + Positional Encoding
**输出端**：线性层 + Softmax → 词表概率分布

#### 2.2 Self-Attention 四步计算
1. **线性投影**：Q = X·W_Q, K = X·W_K, V = X·W_V
2. **计算相似度**：scores = Q·K^T / √d_k
3. **归一化**：weights = softmax(scores)
4. **加权求和**：output = weights · V

**为什么除以 √d_k？**
维度 d_k 越大，点积的方差越大，softmax 后会变得极度尖锐（接近one-hot），梯度变小。除以 √d_k 可以将方差归一化到1，保证训练稳定。

#### 2.3 Multi-Head Attention
将 d_model 拆成 H 个头，每个头维度 d_k = d_model / H，各自独立计算 Attention，最后 concat 再乘 W_O 融合。

- 类比 CNN 的多通道：不同头关注不同模式（语法、语义、位置等）
- 参数量与单头相当，但表达能力显著增强

#### 2.4 Cross Attention（交叉注意力）
**核心特征**：Q 来自一个序列，K、V 来自另一个序列。

**计算方式**：与 Self-Attention 公式完全一致
```
CrossAttn(Q, K, V) = softmax(QK^T / √d_k) · V
```
只是 Q 和 K/V 的来源不同。

**典型应用场景**：
- 机器翻译：Q=目标语言序列，K/V=源语言序列，实现词对齐
- 图像生成（Stable Diffusion）：Q=图像特征，K/V=文本提示词，文本引导图像生成
- 视觉问答（VQA）：Q=问题文本，K/V=图像特征
- DETR 目标检测：Q=目标查询，K/V=图像特征

#### 2.5 现代 LLM 的演变
- **GPT 系列**：砍掉 Encoder 和 Cross-Attention，只保留 Decoder-only 架构
- **LLaMA**：Post-Norm → Pre-Norm，ReLU → SwiGLU，绝对位置编码 → RoPE
- **主流架构**：当前大模型几乎清一色 Decoder-only，Cross-Attention 主要出现在跨模态场景

#### 2.6 高频面试题
**Q：Self-Attention 和 Cross-Attention 的区别？**
A：Self-Attention 的 Q、K、V 都来自同一个序列，用于捕捉序列内部的依赖关系；Cross-Attention 的 Q 来自一个序列，K/V 来自另一个序列，用于实现两个序列之间的信息交互和对齐。

**Q：Transformer 为什么比 RNN 好？**
A：1）并行计算：RNN 必须顺序计算，Transformer 可并行处理所有位置；2）长距离依赖：Attention 直接建立任意两点联系，不受距离衰减影响；3）计算复杂度：序列长度 n 下，Self-Attention 是 O(n²·d)，RNN 是 O(n·d²)，长序列下各有优劣。

**Q：为什么 Decoder 第一层需要 Mask？**
A：因为生成是自回归的，预测第 i 个 token 时只能看到前面的 token，不能看到未来的 token。因果掩码（Causal Mask）通过将上三角位置设为负无穷，softmax 后权重为0，实现信息屏蔽。

### 三、DETR 与 YOLO 篇

#### 3.1 YOLO 核心原理
**核心思想**：将目标检测转化为回归问题，一次前向传播同时预测边界框坐标、置信度和类别概率，实现端到端实时检测。

**关键机制**：
- **网格划分**：图像划分为 S×S 网格，每个网格负责预测中心点落在其中的目标
- **Anchor 锚框**：预设尺寸比例的框，通过 K-means 聚类得到，降低预测难度
- **多尺度预测**：不同特征图检测不同大小目标（大特征图测小目标）
- **NMS 后处理**：非极大值抑制，去除重复检测框

**版本演进趋势**：
- v1-v3：奠定单阶段检测基础，引入多尺度、Anchor
- v5-v8：工程化优化，CSPNet、PANet、更强的骨干网络
- v9-v12：引入注意力机制、无Anchor设计、端到端探索

#### 3.2 DETR 核心原理
**DETR（DEtection TRansformer）** 由 FAIR 于 2020 年提出，首个将 Transformer 成功应用于目标检测的模型。

**核心思想**：将目标检测视为集合预测问题，一次性输出所有目标框，摆脱 NMS 依赖。

**架构组成**：
1. **CNN 骨干网络**：提取图像特征（ResNet等）
2. **Transformer Encoder**：全局注意力建模，增强特征的上下文关联
3. **Transformer Decoder**：接收 N 个可学习的 Object Query，通过 Cross Attention 关注图像特征
4. **预测头**：每个 Query 输出一个框的类别和坐标
5. **二分匹配损失**：匈牙利算法，每个 GT 只匹配一个预测框

#### 3.3 核心对比
| 维度 | YOLO | DETR |
|------|------|------|
| **基础架构** | 单阶段CNN为主 | CNN + Transformer 编码器-解码器 |
| **匹配策略** | 局部一对多，一个GT匹配多个正样本 | 全局一对一，匈牙利算法匹配 |
| **锚框依赖** | 多数版本需要Anchor | 无锚框（Anchor-free） |
| **后处理** | 必须NMS | 无需NMS，端到端 |
| **推理速度** | 快，适合实时场景 | 相对较慢（全局注意力开销大） |
| **小目标检测** | 相对较弱，网格天然短板 | 较强，全局上下文辅助 |
| **遮挡/密集目标** | 易漏检、重复检测 | 更鲁棒，独立查询设计 |
| **训练收敛** | 快，几轮即可见效 | 慢，需要长时间训练 |
| **部署难度** | 低，生态成熟 | 较高，计算量大 |

#### 3.4 RT-DETR 演进
百度提出的 RT-DETR 是 DETR 的实时化版本：
- 采用混合编码器（CNN + Transformer），降低计算量
- 引入 IoU-aware 查询选择
- 速度精度平衡，达到实时检测水准

#### 3.5 高频面试题
**Q：DETR 为什么不需要 NMS？**
A：因为 DETR 通过二分匹配损失（匈牙利算法）训练，强制每个真实目标只匹配一个预测框，模型天然学习到不重复预测。推理时输出固定数量的框，其中包含背景类，直接过滤掉背景即可，无需额外 NMS 后处理。

**Q：YOLO 的正负样本怎么定义？**
A：中心点落在哪个网格，哪个网格就负责预测这个目标。每个网格预测多个Anchor，与GT的IoU最大的那个Anchor为正样本，IoU小于阈值的为负样本，中间的忽略。

**Q：DETR 的 Object Query 是什么？**
A：Object Query 是一组可学习的嵌入向量（通常100个），作为 Decoder 的输入。每个 Query 负责检测一个目标，通过 Cross Attention 从图像特征中"查询"对应目标的信息。它们可以理解为"目标模板"，训练中学习到不同位置、不同形状目标的查询模式。

### Learnable Token 全知识点
#### 一、核心定义与本质
##### 1.1 基本概念
**Learnable Token（可学习Token，又称Soft Token / 可学习嵌入向量）** 是一组独立于输入数据、可随模型端到端训练优化的嵌入向量。它作为输入序列的一部分参与模型计算，不对应任何具体的文本词元、图像块或原始输入信号，其向量值本身就是模型的可训练参数，通过梯度下降不断迭代更新，最终承载特定的任务信息、全局语义或查询意图。

通俗理解：普通Token是模型「读的文字」，可学习Token是模型「自带的思考锚点」，模型自己学着用它完成特定功能。

##### 1.2 与普通Token的核心区别
| 维度 | 普通Token（文本词元/图像Patch） | Learnable Token |
|------|--------------------------------|-----------------|
| 来源 | 由输入数据映射得到（词表索引、图像编码） | 模型自带的可训练参数，与输入无关 |
| 语义 | 对应具体语义（词、图像块） | 无预定义语义，训练中习得功能 |
| 数量 | 随输入长度动态变化 | 固定数量，与输入内容无关 |
| 更新方式 | 词表嵌入随预训练全局更新，推理时固定 | 可针对特定任务单独训练、单独冻结 |
| 典型代表 | 汉字/英文单词、图像Patch特征 | CLS Token、Object Query、Soft Prompt |

##### 1.3 核心价值
1. **全局信息聚合器**：用单个/少量Token聚合整个序列的全局信息，替代人工设计的池化操作
2. **标准化查询接口**：作为Cross Attention的查询端，模板化地从内容序列中提取目标信息
3. **轻量化任务适配**：仅通过少量可学习Token即可适配下游任务，实现参数高效微调
4. **固定长度输出**：无论输入序列多长，最终输出固定数量的Token，方便后续模块处理

#### 二、经典应用场景（面试高频考点）
##### 2.1 NLP领域：BERT  `[CLS]` Token
**定位**：可学习Token最经典的落地，句子级分类任务的标准方案。

###### 工作机制
1. 在输入序列最前端强制插入一个`[CLS]`特殊Token，其嵌入向量为可训练参数，初始无具体语义
2. 经过多层Transformer自注意力计算，`[CLS]`与序列中所有词元交互，自主学习权重分配，逐步聚合整句的全局语义
3. 预训练阶段通过NSP（下一句预测）任务驱动，让模型提前学会用该Token做句子级判断
4. 下游分类任务中，直接取最后一层`[CLS]`的输出向量接入分类头即可

###### 面试核心追问：为什么不用平均池化？
- 平均池化是**等权求和**，无法区分不同词的重要性，信息有损
- `[CLS]`通过注意力机制**自主学习权重分配**，能自动聚焦关键词（如情感词、实体），表达能力更强
- 预训练阶段就与模型联合优化，与下游任务天然适配，迁移效果更好

> 补充：`[SEP]`、`[MASK]`属于词表内的结构特殊Token，虽嵌入也可学习，但归属于词表体系，通常不单独归类为狭义的Learnable Token。

##### 2.2 CV领域两大经典：ViT Class Token + DETR Object Queries
###### （1）ViT Class Token
完全复刻BERT的设计范式：将图像切分为Patch并编码后，在序列开头拼接1个可学习的class token，最终取该token的输出通过MLP完成图像分类。
- 意义：证明了纯Transformer架构在CV领域可完全沿用NLP范式，无需额外的池化或检测头设计
- 地位：Vision Transformer的标志性设计之一

###### （2）DETR Object Queries（目标检测核心考点）
**定位**：最具代表性的多数量可学习Token应用，也是Cross Attention的经典落地，是DETR摆脱Anchor和NMS的核心。

###### 核心原理
1. **本质**：N个（默认100个）d_model维的可学习向量，是模型参数而非图像特征，对所有图片通用
2. **角色**：作为Transformer Decoder的输入，每个Query相当于一个「目标检测槽位」，负责预测一个潜在目标
3. **工作流程**：
   - Decoder自注意力层：Query之间互相交互，避免多个Query预测同一个目标，保证多样性
   - Cross Attention层：每个Query作为Q，从Encoder输出的图像特征（K/V）中检索对应目标的位置和类别信息
   - 最终每个Query输出一个预测框，通过匈牙利二分匹配损失与真实目标一一对应
4. **训练后学到了什么**：不同Query会收敛到负责不同空间位置、不同尺度的目标，隐式学习了目标的空间分布先验，替代了传统检测的人工Anchor设计

##### 2.3 大模型微调：Soft Prompt / Prompt Tuning
**背景**：大模型全量微调成本极高，催生了参数高效微调（PEFT）方向，可学习Token是其中极简的实现路径。

###### 核心思想
冻结整个大模型的全部参数，只在输入序列前拼接K个可学习的Soft Token（又称虚拟Token），仅训练这少量参数来适配下游任务。
- 本质：用可学习Token编码任务指令，引导冻结的大模型转向目标任务，无需修改模型主体
- 参数量级：以7B模型（隐藏层维度4096）为例，20个Soft Token仅约8万参数，远小于LoRA的百万级参数量

###### 关键特性
1. **规模效应**：模型越大效果越好，10B+规模下效果接近全量微调；小模型上表现较差
2. **多任务友好**：一个基座模型可对应多组Soft Prompt，推理时切换仅需替换输入前缀，无需加载多份模型
3. **初始化策略**：通常用相关文本的词嵌入初始化，比随机初始化收敛更快、效果更好

###### 进阶扩展：Prefix Tuning
Prompt Tuning的升级版本：不只在输入层加，而是在**每一层Transformer的注意力K/V前都拼接可学习的Prefix Token**，相当于给每一层都注入任务偏置。
- 表达能力更强，生成类任务效果优于单层Prompt Tuning
- 参数量有所增加，但仍远小于全量微调

##### 2.4 多模态场景
1. **模态占位Token**：多模态模型中的`<image>`、``等标记，部分设计中其初始嵌入为可学习，用于占位并引导模型识别模态边界，推理时会被替换为对应模态的特征序列
2. **可学习查询Token**：VQA、图文检索等模型中，用固定数量的可学习Query聚合图像特征，再与文本问题交互，统一多模态交互接口

#### 三、关键技术对比（面试高频辨析题）
##### 3.1 可学习CLS Token vs 全局平均池化（GAP）
| 维度 | 可学习CLS Token | 全局平均池化 |
|------|----------------|--------------|
| 聚合方式 | 注意力加权，模型自主学习权重分配 | 等权平均，人工固定规则 |
| 表达能力 | 强，可捕捉关键语义信息 | 弱，信息压缩有损 |
| 可训练参数 | 有，端到端优化 | 无，固定操作 |
| 计算开销 | 略高（多一个Token的注意力计算） | 极低 |
| 适用场景 | 复杂分类、语义理解、高精度任务 | 轻量模型、简单分类任务 |

##### 3.2 Prompt Tuning（可学习Token） vs LoRA（PEFT两大主流方案）
| 维度 | Prompt Tuning | LoRA |
|------|--------------|------|
| 核心原理 | 输入层加可学习Token，从外部注入任务信息 | 权重矩阵加低秩更新，修改模型内部计算 |
| 参数量 | 极少（万级） | 较少（十万~百万级） |
| 作用位置 | 仅输入嵌入层 | 注意力层、FFN层等内部权重 |
| 大模型效果 | 好，模型越大越接近全量微调 | 好，全尺寸模型上都稳定 |
| 小模型效果 | 差，几乎不可用 | 依然有效 |
| 任务切换 | 直接替换输入前缀，零成本切换 | 需要权重合并/动态挂载 |
| 推理开销 | 输入长度略有增加，几乎无额外成本 | 合并后零额外开销 |
| 适合场景 | 多任务批量部署、快速适配简单任务 | 复杂任务、追求效果上限 |

##### 3.3 可学习Query vs 人工设计Anchor
| 维度 | 可学习Query（DETR） | 人工Anchor（YOLO） |
|------|---------------------|--------------------|
| 设计方式 | 数据驱动，端到端自动学习 | 人工统计数据集设计尺寸比例 |
| 先验依赖 | 无人工先验，泛化性强 | 强依赖人工经验，适配新数据集需重新设计 |
| 收敛速度 | 慢，需要大量迭代 | 快，初始化合理 |
| 密集目标 | 更鲁棒，Query独立不冲突 | 易出现重复检测，需NMS后处理 |
| 工程复杂度 | 高，Transformer训练难度大 | 低，生态成熟 |

#### 四、高频面试题
**Q：什么是Learnable Token？和普通词元有什么区别？**
A：Learnable Token是一组不对应具体输入内容、可随模型端到端训练的嵌入向量，本身就是模型的可训练参数。它和普通Token的核心区别在于：普通Token由输入数据映射得到，承载具体语义，数量随输入变化；可学习Token是模型自带的参数，数量固定，在训练中习得特定功能（如全局聚合、目标查询），不依赖输入内容。

**Q：BERT为什么用CLS token做分类，不用所有token的平均池化？**
A：核心原因是表达能力更强。平均池化是等权求和，无法区分不同词的重要性，会丢失关键信息；CLS token通过自注意力机制自主学习权重分配，能自动聚焦序列中的关键词义，更适合复杂语义任务。同时预训练阶段的NSP任务已经让CLS token学习了句子级表征能力，下游迁移效果更好。

**Q：DETR的Object Query是什么？它是怎么工作的？**
A：Object Query是DETR中一组可学习的嵌入向量（默认100个），作为Decoder的输入，每个Query对应一个目标检测槽位。工作流程上，Query先通过自注意力互相协调避免重复，再通过Cross Attention从图像特征中检索目标信息，最终每个Query输出一个预测框。训练后，不同Query会分工负责不同位置、不同大小的目标，隐式替代了传统检测的Anchor设计。

**Q：Prompt Tuning的原理是什么？为什么少量可学习token就能适配任务？**
A：Prompt Tuning的核心是冻结大模型全部参数，仅在输入层加入少量可学习的Soft Token，通过训练这些Token来引导大模型输出目标任务的结果。之所以少量参数就有效，是因为大模型在预训练阶段已经掌握了通用语言能力和世界知识，下游任务只需要引导模型调用对应能力，不需要重新学习知识，可学习Token本质就是编码了任务的「隐形指令」。

**Q：可学习Token需要加位置编码吗？**
A：需要。可学习Token只是内容向量可学习，它在序列中占据固定的位置，同样需要位置信息来和其他Token建立位置关联。比如CLS Token在序列首位，会叠加对应位置的位置编码；DETR的Object Query也会搭配可学习的位置编码共同使用。

**Q：Soft Prompt的数量怎么选？越多越好吗？**
A：不是越多越好。数量太少表达能力不足，无法编码复杂任务；数量太多容易过拟合，还会增加输入长度、提升推理成本。通常5~100个是常用区间：简单分类任务个位数即可，复杂生成任务可用几十到上百个。
